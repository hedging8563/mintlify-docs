Les sorties structurées et les outils sont de véritables chemins d'API AI Sonar, mais ils ne constituent pas une capacité universelle à travers chaque modèle et route. Choisissez la route qui possède le comportement dont vous avez besoin.

## JSON Et Sortie Structurée

Pour des réponses structurées portables, commencez par les Complétions de Chat `response_format: {"type":"json_object"}` et validez la sortie dans votre application. C'est la forme de demande publique stable dans la validation partagée de AI Sonar.

```json
{
  "model": "gpt-5.4",
  "messages": [{"role": "user", "content": "Return a JSON object with city and weather."}],
  "response_format": {"type": "json_object"}
}
```

Les chemins de sortie structurée `json_schema` et `strict` existent dans certains codes en amont et de conversion, mais ils dépendent du modèle et de la route. Traitez-les comme une capacité à vérifier avec le modèle sélectionné, et non comme une promesse générale à travers chaque route AI Sonar.

## Limite d'Appel d'Outils

| Forme d'outil | Route à privilégier | Remarque de production |
|----------------|---------------------|-----------------------|
| Outils de fonction OpenAI | `/v1/chat/completions` | Portable pour les appels de fonction définis par l'utilisateur |
| Outils de réponses OpenAI | `/v1/responses` | À utiliser pour les outils intégrés ou hébergés spécifiques aux réponses |
| Outils Anthropic | `/v1/messages` | À utiliser pour l'utilisation d'outils natifs Claude et les flux de réflexion |
| Déclarations de fonction Gemini / outils intégrés | `/v1beta/models/:model:generateContent` | Conservez les champs d'outil natifs Gemini sur la route Gemini |

AI Sonar n'exécute pas votre fonction pour vous. Lorsque un modèle retourne `tool_calls`, `function_call`, `tool_use`, ou des parties d'appel de fonction Gemini, votre application doit exécuter l'outil, ajouter le résultat de l'outil dans le format requis par la route, et continuer la conversation.

## Pièges Courants

- Ne pas envoyer d'outils intégrés natifs au fournisseur via les Complétions de Chat et s'attendre à ce qu'ils soient traduits silencieusement.
- Ne pas compter sur le schéma JSON strict à moins que la route et le modèle sélectionnés aient été testés.
- Gardez les schémas petits et explicites ; de grands schémas ajoutent des tokens et peuvent réduire la fiabilité.
- Validez la sortie du modèle côté serveur même lorsque le mode JSON est activé.
- Traitez les erreurs de champ non pris en charge comme un retour d'information sur le contrat. Supprimez le champ ou utilisez la route native qui le possède.

## Exemple

```bash
curl https://api.aisonar.dev/v1/chat/completions \\
  -H "Authorization: Bearer sk-your-api-key" \\
  -H "Content-Type: application/json" \\
  -d '{
    "model": "gpt-5.4",
    "messages": [{"role":"user","content":"Extract name and email as JSON."}],
    "response_format": {"type":"json_object"},
    "tools": [{
      "type": "function",
      "function": {
        "name": "lookup_customer",
        "description": "Look up a customer by email",
        "parameters": {
          "type": "object",
          "properties": {"email": {"type": "string"}},
          "required": ["email"]
        }
      }
    }]
  }'
```
