AI Sonar image APIs cover several model families with different request shapes. Use the public model contract and the endpoint-specific API reference before assuming a field works across every image model.

## Choose The Endpoint

| Goal | Endpoint | Notes |
|------|----------|-------|
| Text-to-image | `POST /v1/images/generations` | Use `prompt` and an explicit `model` |
| Image-to-image / reference image | `POST /v1/images/generations` | Use `operation: "image-to-image"` plus `image_url`, `image_urls`, or `reference_image_urls` when the model supports it |
| GPT Image edit | `POST /v1/images/edits` | Supports multipart `image`, JSON `image_url` / `image_urls`, and official `images[]` references for supported edit models |
| Legacy variation | `POST /v1/images/variations` | Prefer current generation or edit routes for new work |
| Async status | `GET /v1/tasks/{id}` | Use `poll_url` first when present |

## Model Discovery

```bash
curl "https://api.aisonar.dev/v1/models?recommended_for=image" \\
  -H "Authorization: Bearer sk-your-api-key"
```

Image model parameters differ a lot. Read the selected model contract before sending `size`, `quality`, `aspect_ratio`, `resolution`, `async`, `background`, or `output_format`.

## Inputs And Model-Specific Fields

- `gpt-image-2` uses OpenAI-style `size` and `quality`; omit `size` or `quality` to let AI Sonar use `auto` where supported.
- Gemini and Nano Banana image families generally use `aspect_ratio`; only send `resolution` when the selected model exposes it.
- Nano Banana reference-image requests belong on `/v1/images/generations` with `operation: "image-to-image"` and `image_urls`, not on `/v1/images/edits`.
- `/v1/images/generations` does not accept top-level `images[]` or `file_id`; those are edit-flow shapes.
- Remote image URLs must be public `http`/`https` URLs. Avoid private networks, embedded credentials, fragments, and short-lived signed URLs.

## Create Request

```bash
curl https://api.aisonar.dev/v1/images/generations \\
  -H "Authorization: Bearer sk-your-api-key" \\
  -H "Content-Type: application/json" \\
  -d '{
    "model": "gpt-image-2",
    "prompt": "A clean product photo of a ceramic coffee cup on a walnut desk",
    "size": "1024x1024",
    "response_format": "url"
  }'
```

## Polling Results

Some image routes return inline `data[].url` or `data[].b64_json`. Other routes return `status: "pending"`, `task_id`, and `poll_url`. If you request async image generation or editing, completed task results return URLs; use synchronous requests when you specifically need `b64_json`.

```bash
curl "https://api.aisonar.dev/v1/tasks/$TASK_ID" \\
  -H "Authorization: Bearer sk-your-api-key"
```

## Common Pitfalls

- Do not assume every image model is synchronous.
- Do not request `b64_json` from async image tasks; async results are URL-oriented.
- Do not send `resolution` to ratio-only Gemini image models.
- Do not use `/v1/images/edits` for Nano Banana image-to-image flows.
- Set HTTP client timeouts to at least `120s` for synchronous high-resolution image requests.
