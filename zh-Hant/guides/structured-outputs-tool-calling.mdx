結構化輸出和工具是真正的 AI Sonar API 路徑，但它們並不是每個模型和路徑都具備的通用能力。選擇擁有您所需行為的路徑。

## JSON 與結構化輸出

對於可攜式結構化響應，請從 Chat Completions 開始 `response_format: {"type":"json_object"}` 並在您的應用程序中驗證輸出。這是 AI Sonar 共享驗證中的穩定公共請求形狀。

```json
{
  "model": "gpt-5.4",
  "messages": [{"role": "user", "content": "Return a JSON object with city and weather."}],
  "response_format": {"type": "json_object"}
}
```

`json_schema` 和 `strict` 結構化輸出路徑存在於某些上游和轉換代碼中，但它們依賴於模型和路徑。將它們視為與所選模型驗證的能力，而不是在每個 AI Sonar 路徑上的普遍承諾。

## 工具調用邊界

| 工具形狀 | 優先路徑 | 生產備註 |
|----------|----------|----------|
| OpenAI 函數工具 | `/v1/chat/completions` | 可攜式用於用戶定義的函數調用 |
| OpenAI 響應工具 | `/v1/responses` | 用於響應特定的內置或托管工具 |
| Anthropic 工具 | `/v1/messages` | 用於 Claude 原生工具使用和思考流程 |
| Gemini 函數聲明 / 內置工具 | `/v1beta/models/:model:generateContent` | 在 Gemini 路徑上保留 Gemini 原生工具字段 |

AI Sonar 不會為您執行函數。當模型返回 `tool_calls`、`function_call`、`tool_use` 或 Gemini 函數調用部分時，您的應用程序必須運行該工具，將工具結果附加到路徑所需的格式中，並繼續對話。

## 常見陷阱

- 不要通過 Chat Completions 發送提供者原生的內置工具，並期望它們被靜默翻譯。
- 除非所選路徑和模型已經過測試，否則不要依賴嚴格的 JSON Schema。
- 保持 schema 小而明確；大型 schema 會增加 tokens 並可能降低可靠性。
- 即使啟用了 JSON 模式，也要在伺服器端驗證模型輸出。
- 將不支持的字段錯誤視為合約反饋。刪除該字段或使用擁有該字段的原生路徑。

## 範例

```bash
curl https://api.aisonar.dev/v1/chat/completions \\
  -H "Authorization: Bearer sk-your-api-key" \\
  -H "Content-Type: application/json" \\
  -d '{
    "model": "gpt-5.4",
    "messages": [{"role":"user","content":"Extract name and email as JSON."}],
    "response_format": {"type":"json_object"},
    "tools": [{
      "type": "function",
      "function": {
        "name": "lookup_customer",
        "description": "Look up a customer by email",
        "parameters": {
          "type": "object",
          "properties": {"email": {"type": "string"}},
          "required": ["email"]
        }
      }
    }]
  }'
```
